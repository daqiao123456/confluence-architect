# 三、器（工具承载）

Confluence 是主器。承载空间、页面树、模板、导航、归档。

辅器分工如下：

- **标签（Label）**：做横向索引和交叉定位
- **模板（Template）**：保证文档结构一致
- **搜索（Search）**：做统一检索入口
- **权限（Permission）**：控制可见与可编辑边界
- **归档（Archive）**：防止过时内容污染主区

器只负责承载与执行。
器不决定分类逻辑。分类逻辑由"事—文—位"决定。

# 九、定治理——治理规则

| 维度 | 规则 |
|------|------|
| Owner | 每篇文档有且只有 1 个个人 Owner。默认由对应对象组负责人指定；制度类文档归部门负责人指派；运行类文档归值班或服务负责人指派。禁止"某团队共同负责"。 |
| 生命周期 | 创建：作者发起；审核：对象组 Owner 或指定评审人审核；发布：Owner 确认状态改为 active；更新：变更发起人或 Owner 触发；归档：对象组 Owner 在失效、替代、项目结束后触发；删除：仅空间管理员按归档期满规则执行。 |
| 审查节奏 | 高风险内容如安全策略、API 文档、权限矩阵、计费相关接口，30-60 天；中风险内容如核心流程、架构规范、产品规格，90 天；低风险内容如会议记录、团队笔记，180 天；长青概念类内容年度审查。以变更驱动为主，日历驱动为辅。 |
| 质量门槛 | 准入最低要求：标题清晰、Owner 明确、适用范围明确、最近更新时间可追溯、标签四维齐全、正文用模板、操作类必须含前置条件/步骤/回滚、参考类必须含字段或规则定义、解释类必须说明原理与适用边界。 |
| 权限模型 | 默认全员可读；对象组成员可编辑本域正文；部门知识管理员可编辑导航与标签规范；项目交付区仅项目成员可编辑；复盘归档区只读；删除权限仅空间管理员持有；高敏安全页按最小权限单独授权。 |

# 十、度量——运营指标

|------|------|------|
| 平均查找时间 | 技术人员从提出问题到定位目标文档的平均耗时 | 是否找得到，是否能在可接受时间内找到 |
| 文档覆盖率 | 技术部关键对象组应有文档与实际已建文档的比例 | 是否覆盖完整职责 |
| 内容新鲜度 | 超过审查周期未更新的 active 文档占比 | 是否过期、是否腐烂 |
| 搜索成功率 | 用户一次搜索后直接命中目标文档的比例 | 搜索是否准确、标签是否有效 |

建议基线如下：

| 指标 | 衡量方式 | 建议基线值 |
|------|------|------|
| 平均查找时间 | 抽样 20 个高频任务，如发布、权限申请、故障升级、扩容、漏洞修复，记录首次命中文档耗时中位数 | ≤ 3 分钟 |
| 文档覆盖率 | 以 29 个对象组为盘点清单，检查四类主文档是否至少覆盖 Reference + How-to，两者齐全计覆盖 | ≥ 90% |
| 内容新鲜度 | 统计 active 文档中超出审查周期的比例；高风险按 60 天、中风险按 90 天、低风险按 180 天 | 过期占比 ≤ 10% |
| 搜索成功率 | 抽样 50 个真实搜索词，统计一次搜索前 5 条结果内命中目标文档的比例 | ≥ 85% |

一句话总结：技术部知识库不是先搭目录，而是先按稳定职责定骨架，再按知识用途定文档，再按使用场景定入口。
